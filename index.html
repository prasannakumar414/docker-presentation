Docker Timeline

From dotCloud to the container revolution

2008

dotCloud Founded

Solomon Hykes founds dotCloud, a PaaS company — Docker's predecessor.

2013

Docker Open-Sourced

Docker is released as open-source at PyCon. The container revolution begins.

2014

Docker Hub & 1.0

Docker Hub launches. Docker 1.0 released — declared production-ready.

2015

OCI Founded

Open Container Initiative established. Docker Compose and Docker Swarm released.

2016

containerd Donated

Docker donates containerd to the CNCF. Swarm mode built into Docker Engine.

2017

Moby Project

Docker open-sources its platform as the Moby Project. Multi-stage builds arrive.

2020

Docker Desktop Changes

Kubernetes becomes Kubernetes. Docker Desktop adopts a new subscription model.

2024

Docker Today

20M+ developers, 300B+ image pulls. Docker Build Cloud and AI integrations.

Linux Cgroups & Namespaces

The two pillars that make containers possible

⚙️

Control Groups (cgroups)

Limit, account for, and isolate the resource usage (CPU, memory, disk I/O, network) of process groups.

CPU Memory Block I/O Network Devices
🔒

Namespaces

Partition kernel resources so each set of processes sees its own isolated instance of the global resource.

Docker CLI Commands

Your interface to the Docker ecosystem

docker run

Create and start a container from an image

$ docker run -d -p 8080:80 --name web nginx:latest
docker build

Build an image from a Dockerfile

$ docker build -t myapp:v1 .
docker pull

Download an image from a registry

$ docker pull python:3.12-slim
docker ps

List running containers

$ docker ps --format "table {{.Names}}\t{{.Status}}"
docker exec

Execute a command in a running container

$ docker exec -it web /bin/bash
docker compose

Define and run multi-container applications

$ docker compose up -d --build

Inside dockerd

The Docker Daemon — the brain of Docker

📨

REST API Server

Listens on unix:///var/run/docker.sock. Receives requests from the CLI.

🖼️

Image Management

Pull, build, tag, and push images. Manages local image store and layer cache.

🌐

Network Management

Creates bridge, overlay, and macvlan networks. Manages container DNS and IP allocation.

💾

Volume Management

Creates and mounts volumes. Supports local, NFS, and plugin-based drivers.

📦

Delegates to containerd

Sends gRPC calls to containerd for actual container lifecycle management.

🛡️ containerd-shim

Acts as the parent process of the container. Allows dockerd/containerd to restart without killing running containers.

📋 Responsibilities

Keeps STDIO open, reports exit status, allows daemonless containers. One shim per container.

🔄 Decoupling

Docker daemon can be upgraded/restarted without disrupting running workloads.

runc & OCI Runtimes

The low-level container runtimes that create processes

runc

The reference OCI runtime. Spawns containers using Linux namespaces and cgroups directly.

Default · Go

crun

Written in C for faster startup. OCI-compliant, drop-in replacement for runc.

Fast · C

Kata Containers

Runs containers in lightweight VMs. Combines the security of VMs with container speed.

Secure · VMs

gVisor (runsc)

Google's application kernel. Intercepts syscalls for an extra isolation layer.

Secure · Sandbox

All runtimes implement the OCI Runtime Specification — a standard interface for creating and running containers, ensuring interoperability.

Writing a Dockerfile

Build your image instruction by instruction

Dockerfile
1# --- Build stage ---
2FROM golang:1.22-alpine AS builder
Multi-stage build: first stage compiles the Go binary
3WORKDIR /app
Set the working directory inside the build container
4COPY go.mod go.sum ./
Copy module files first to leverage Docker layer caching
5RUN go mod download
Download dependencies — cached layer if go.mod hasn't changed
6COPY . .
Copy the rest of the application source code
7RUN CGO_ENABLED=0 go build -o server .
Compile a static binary — CGO_ENABLED=0 for a fully static build
8# --- Runtime stage ---
9FROM alpine:3.19
Tiny runtime image — no Go toolchain, just the binary
10WORKDIR /app
11COPY --from=builder /app/server .
Copy only the compiled binary from the build stage
12EXPOSE 8080
Document the port the app listens on (metadata only)
13USER nobody
Run as non-root user for security
14CMD ["./server"]
Run the compiled Go binary when the container starts

Dockerfile Best Practices

Hover or click cards to reveal the details

📦

Use Multi-Stage Builds

Reduce final image size

✅ Do

Separate build and runtime stages. Copy only artifacts you need into the final image.

🪶

Minimize Layers

Combine RUN commands

❌ Don't

Don't use separate RUN for each command. Chain with && and use line continuations.

🏔️

Use Alpine/Slim Base

Start small, stay small

✅ Do

Use -alpine or -slim variants. Reduces attack surface and image size dramatically.

📄

Use .dockerignore

Exclude unnecessary files

✅ Do

Add vendor/, .git, *.log, and secrets to .dockerignore to speed up builds and reduce context.

🔒

Don't Run as Root

Security first

✅ Do

Use the USER instruction to switch to a non-root user. Limits damage if the container is compromised.

💎

Leverage Build Cache

Order matters

✅ Do

Copy dependency files before source code. Least-changing layers first, most-changing layers last.

Container Security

Reduce your attack surface and harden your containers

✅ Security Checklist

✓

Use official and verified base images

✓

Scan images for vulnerabilities (Trivy, Snyk)

✓

Run containers as non-root user

✓

Use read-only filesystems where possible

✓

Drop unnecessary Linux capabilities

✓

Keep images minimal — no shells, no compilers

✓

Use Docker Content Trust for image signing

✓

Never store secrets in images or env vars
